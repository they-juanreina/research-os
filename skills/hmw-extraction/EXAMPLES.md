# HMW Extraction Examples

These examples show complete HMW statements in the standard output format. Use these as templates for your own extraction work.

---

## Example 1: Navigation & Findability

| Field | Value |
|-------|-------|
| **HMW_ID** | HMW-01 |
| **Theme** | Navigation & Findability |
| **Personas** | Content managers, Site editors |
| **Pain Point** | Editors cannot quickly locate existing content before creating duplicates; they manually search through three different archives and category trees. |
| **Current Workaround** | Keep a shared spreadsheet of recent content; bookmark important pages; ask colleagues if they've seen similar content. |
| **HMW Statement** | How might we help content editors locate related existing content before starting a new piece, so they can build on previous work instead of duplicating effort? |
| **Priority** | High |
| **Evidence** | "I always ask the team if we've covered this before, but nobody remembers." (Interview P2); 5 of 6 editors reported re-creating similar content at least once per month. |
| **Design Implications** | Smart content suggestion system; contextual search at point of creation; related content sidebar; content tagging and discovery interface. |
| **Confidence** | High |

## Example 2: Trust & Security

| Field | Value |
|-------|-------|
| **HMW_ID** | HMW-02 |
| **Theme** | Trust & Security |
| **Personas** | System administrators, Compliance officers |
| **Pain Point** | Administrators cannot verify who currently has access to sensitive systems; audit logs are dense and scattered across multiple tools, making it hard to prove compliance during reviews. |
| **Current Workaround** | Export access lists to spreadsheets; manually cross-reference logs; spend hours before compliance audits assembling proof of access controls. |
| **HMW Statement** | How might we help system administrators quickly generate an accurate snapshot of current access permissions without leaving the platform, so they can confidently demonstrate compliance controls? |
| **Priority** | High |
| **Evidence** | "Compliance audits terrify me because I have to manually reconstruct everything." (Interview P8); Average audit prep time: 16 hours per administrator. |
| **Design Implications** | Real-time access dashboard; role-based access reports; audit trail visualization; compliance-ready export formats; access history timeline. |
| **Confidence** | High |

## Example 3: Terminology & Mental Models

| Field | Value |
|-------|-------|
| **HMW_ID** | HMW-03 |
| **Theme** | Terminology & Mental Models |
| **Personas** | New users, Occasional users |
| **Pain Point** | New users are confused by system terminology that doesn't match their domain language; they abandon workflows because labels like "matrix," "payload," and "transformation" feel technical and intimidating. |
| **Current Workaround** | Reach out to support; ask colleagues; guess and try multiple times; skim documentation; watch YouTube tutorials. |
| **HMW Statement** | How might we help new users understand system concepts using their own domain language, so they feel confident navigating core workflows without external help? |
| **Priority** | High |
| **Evidence** | "I don't know if 'transformation' means I'm changing the data or just the format." (Interview P3); 40% of new users contact support within first week about terminology. |
| **Design Implications** | Contextual tooltips with domain-specific translations; glossary with examples; progressive disclosure of terminology; in-context micro-learning; role-based onboarding language. |
| **Confidence** | Medium |

## Example 4: Permissions & Access Control

| Field | Value |
|-------|-------|
| **HMW_ID** | HMW-04 |
| **Theme** | Permissions & Access Control |
| **Personas** | Team leads, Project owners |
| **Pain Point** | Team leads cannot grant specific permissions without giving access to sensitive areas they don't intend to expose; granular permission interfaces are hidden three levels deep, forcing leaders to request help from administrators for simple access changes. |
| **Current Workaround** | File a support ticket; ask administrators to handle access changes; give broad permissions and trust team members to self-regulate; maintain external permission spreadsheets. |
| **HMW Statement** | How might we help team leads grant specific permissions to team members at point of need, without navigating complex permission hierarchies or asking for administrator help? |
| **Priority** | Medium |
| **Evidence** | "It takes me three days to get someone the access they need because I have to contact IT." (Interview P5); 12 support tickets per week for permission-related requests. |
| **Design Implications** | Simplified permission UI at team level; context-aware permission suggestions; permission templates for common roles; bulk permission changes; permission request and approval workflow. |
| **Confidence** | Medium |

## Example 5: Automation & Repetitive Tasks

| Field | Value |
|-------|-------|
| **HMW_ID** | HMW-05 |
| **Theme** | Automation & Repetitive Tasks |
| **Personas** | Operations analysts, Data processors |
| **Pain Point** | Analysts spend 3+ hours per week on repetitive data entry and formatting tasks; they copy data between systems, reformat spreadsheets, and manually verify matching records—work that could be automated but requires technical skills they don't have. |
| **Current Workaround** | Use desktop automation tools like AutoHotkey or Zapier; write simple scripts; accept manual work as part of the job; hire temporary contractors for data processing; batch work and do it in bulk to save context switching. |
| **HMW Statement** | How might we help operations analysts automate repetitive data tasks without writing code, so they can focus on analysis and decision-making instead of manual data handling? |
| **Priority** | High |
| **Evidence** | "I spend all day moving data between systems." (Interview P7); Time tracking showed 18 hours per week on data entry for a 40-hour role; 100% of interviewed analysts expressed frustration with manual data work. |
| **Design Implications** | No-code automation builder; workflow templates for common tasks; record and replay automation; integration with external systems; smart data mapping and validation; automation scheduler and monitoring. |
| **Confidence** | High |

## How to Use These Examples

1. **Match your pain point to a theme:** Use the five themes (Navigation, Trust, Terminology, Permissions, Automation) as starting points for clustering your own research.

2. **Follow the structure:** Copy the table format and fill in your fields systematically. Don't skip Evidence or Design Implications.

3. **Test scope:** For each HMW statement, ask: "Can I think of 3-5 different design solutions?" If yes, it's ready. If no, rewrite to be less prescriptive.

4. **Prioritize by combining scores:** Use Impact × Urgency / Feasibility to rank these HMW statements against each other.

5. **Evidence is non-negotiable:** Every HMW statement must have research backing. If you can't cite evidence, revisit your research or reframe the opportunity.

## Template for Your Own Extractions

Copy and complete this table structure for each pain point you identify:

| Field | Value |
|-------|-------|
| **HMW_ID** | HMW-XX |
| **Theme** | [Navigation / Trust / Terminology / Permissions / Automation / Other] |
| **Personas** | [List specific roles or personas] |
| **Pain Point** | [Describe the friction or barrier in 1-2 sentences] |
| **Current Workaround** | [How do users currently cope?] |
| **HMW Statement** | How might we help [PERSONA] [GOAL/BENEFIT] [CONTEXT]? |
| **Priority** | [High / Medium / Low] |
| **Evidence** | [Direct quote or observation] |
| **Design Implications** | [2-4 potential solution directions] |
| **Confidence** | [High / Medium / Low] |

Fill out each row completely. The evidence and design implications fields prevent vague HMW statements and keep ideation grounded in research.
